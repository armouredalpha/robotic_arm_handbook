Welcome to Your MoveIt Guidebook

This is your step-by-step guide to learning MoveIt with the Panda robotic arm. Follow along with each section to explore RViz and motion planning hands-on.

Hands-On: Interact with Your Robotic Arm

Project 'O'-Exploring rviz

In this section, you will explore RViz and get familiar with the MoveIt environment before diving into motion planning.

Step 1: Launch the Demo

Start the MoveIt demo with RViz:

cd ~/ws_moveit2
source install/setup.bash
unset GTK_PATH
export LD_LIBRARY_PATH=$(echo "$LD_LIBRARY_PATH" | tr ':' '\n' | grep -v snap | tr '\n' ':')
ros2 launch moveit2_tutorials demo.launch.py rviz_config:=panda_moveit_config_demo_empty.rviz

You will see an empty RViz world. You now need to add the Motion Planning plugin.

Step 2: Add the Motion Planning Plugin

  1. In the RViz Displays tab, click Add.
  2. From the moveit_ros_visualization folder, select MotionPlanning as the Display Type and click OK.
  3. The Panda robot arm should now appear in the viewport.

Step 3: Configure the Plugin

Click on MotionPlanning in the Displays panel and configure the following:

  • In Global Options, set the Fixed Frame to /panda_link0
  • In Planning Request, change the Planning Group to panda_arm

These two settings are essential — they tell MoveIt which coordinate frame and which joint group to plan for.

Step 4: Set Up Interactive Markers

Enable both the start and goal state markers so you can interact with the arm:

  1. Under Planning Request, check Query Start State — this shows a green arm representing where the robot begins.
  2. Under Planning Request, check Query Goal State — this shows an orange arm representing where the robot should move to.
  3. If you do not see the drag handles, click Interact in the RViz top menu (add it via the + button if missing).

You can now drag both arms to any position in the workspace. Try moving them around!

Step 5: Explore Collision Detection

MoveIt can detect when links collide with each other in real time.

  1. Uncheck Query Goal State temporarily so only the green (start) arm is visible.
  2. Drag the arm until two links overlap — they will turn red, indicating a collision.
  3. Now check Use Collision-Aware IK in the Planning tab. Try again — the solver will now try to find a valid pose that avoids collision instead.
Key insight: When Collision-Aware IK is on, the robot refuses impossible or unsafe poses. This is how real robots avoid self-damage.

Step 6: Plan a Motion

Now plan an actual trajectory between your start and goal states:

  1. Re-enable Show Robot Visual under Planned Path and check Show Trail.
  2. Re-enable Query Goal State.
  3. Drag the green arm (start) and orange arm (goal) to different positions. Make sure neither is in a collision state.
  4. In the MotionPlanning panel, click the Plan button.
  5. Watch the arm animate through the planned path, with a trail showing the end-effector trajectory.

Step 7: Try Cartesian Planning

By default, MoveIt plans in joint space. Enabling Cartesian mode forces the end-effector to move in a straight line.

  1. In the Planning tab, check Use Cartesian Path.
  2. Move the goal state and click Plan again.
  3. Observe how the motion is now a direct straight-line path vs. the curved joint-space motion from before.
This is useful in tasks like welding or painting where the tool must follow a precise linear path.

Step 8: Execute & Adjust Speed

Click Plan & Execute to send the trajectory to the simulated robot. The robot will move in the simulation.

The default velocity and acceleration are set to 10% of the robot's maximum. You can increase these using the speed sliders in the Planning tab — but move carefully when increasing speed in a real robot context.

Bonus: Inspect Waypoints with the Trajectory Slider

  1. From the Panels menu, select Trajectory > Trajectory Slider.
  2. Plan a motion, then use the Slider panel to step through each waypoint of the trajectory one by one.
  3. Click Play to animate through them automatically.
This is a great debugging tool to understand exactly how MoveIt breaks a motion into discrete steps.

Save Your RViz Configuration

Once you are happy with your setup, save it so you don't have to reconfigure next time:

  1. In RViz, go to File > Save Config As.
  2. Save to moveit2_tutorials/doc/tutorials/quickstart_in_rviz/launch/ and rebuild the workspace.
  3. Load it next time with:
ros2 launch moveit2_tutorials demo.launch.py rviz_config:=your_rviz_config.rviz

Part 2 - Inverse Kinematics

Pick & Place with MoveIt Task Constructor

Hands-On Code Walkthrough

What is MoveIt Task Constructor (MTC)?

In Part 1, you planned a single motion — move the arm from point A to point B. But real robot tasks are made of many steps: open the gripper, approach the object, grasp it, lift it, move to a new location, and place it down.

MoveIt Task Constructor (MTC) lets you define these multi-step tasks as a sequence of stages. Each stage is responsible for one small piece of the overall plan, and MTC connects them together automatically.

Think of it like a recipe: each stage is one instruction. MTC figures out how to connect each step to the next so the whole plan works together.

Launch & Observe MTC in Action

Open two terminals. In Terminal 1, launch the MoveIt environment and RViz:

cd ~/ws_moveit2
 
source install/setup.bash
unset GTK_PATH
export LD_LIBRARY_PATH=$(echo "$LD_LIBRARY_PATH" | tr ':' '\n' | grep -v snap | tr '\n' ':')
ros2 launch moveit2_tutorials mtc_demo.launch.py

In Terminal 2, launch your pick and place node:

cd ~/ws_moveit2
 
source install/setup.bash
unset GTK_PATH
export LD_LIBRARY_PATH=$(echo "$LD_LIBRARY_PATH" | tr ':' '\n' | grep -v snap | tr '\n' ':')
 ros2 launch mtc_tutorial pick_place_demo.launch.py

Reading the Terminal Output

When MTC runs, it prints a stage diagram to the terminal. Here is how to read it:

1  - ←   1 →   -  0 / initial_state
-  0 →   0 →   -  0 / move_to_home
  • The numbers to the left/right of the arrows show how many solutions were propagated backward / forward.
  • The number between the arrows shows how many solutions were generated by the stage.
  • A 0 between the arrows means the stage failed to find a solution — this is the stage to debug.
  • If a stage shows 0 solutions propagated forward, check the stage before it for the root cause.
Common failure: if planning fails, try increasing setTimeout() on your Connect stages, or check that the cylinder is within the robot's reach by adjusting its position in setupPlanningScene().

Quick Reference: Stage Types

Stage Type Example What it does
Generator CurrentState, GenerateGraspPose Creates starting points for planning; no input needed
Propagator MoveRelative, MoveTo Takes a state and extends it; moves the robot
Connector Connect Bridges two stages by planning the motion between them
Container SerialContainer Groups multiple stages together as one logical unit
